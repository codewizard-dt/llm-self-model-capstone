## Why this slice exists

`package-scaffold` set up a **repo-root uv workspace** with a single shared `.venv`. That is the
wrong model for this repo: the authoritative source (MASTER_REQUIREMENTS â†’ Tech stacks) pins
**different Python versions per vertical** â€” `contracts` & `operator` on **3.12**, `pilot` &
`coprocessor` on **3.11**. A uv workspace has exactly one shared environment / one interpreter, so it
cannot host 3.11 and 3.12 at once. The workspace only *looks* fine today because `contracts` is the
only stack that exists. This slice replaces the shared-interpreter model with **independent per-stack
uv projects**, each pinning its own Python and owning its own `.venv` â€” matching the conventions'
"each vertical is its own Stack/Factory."

`package-scaffold` is a completed (immutable) slice; this is the sanctioned forward correction.

## What this slice delivers

### 1. Remove the root uv workspace
- Delete the `[tool.uv.workspace]` stanza from the root `pyproject.toml`. The root no longer binds
  the stacks into one environment.
- Delete the root `uv.lock` (it was the workspace lock). Each stack now owns its own lock under its
  own root (e.g. `contracts/uv.lock`, generated by `cd contracts && uv sync`).
- If the root `pyproject.toml` has no remaining purpose after removing the workspace, delete it too.
  (Decide during implementation: keep ONLY if something at repo root genuinely needs it â€” it should
  not, once the build gate is routed through `make`.)

### 2. `contracts/` becomes a fully standalone uv project
- `contracts/.python-version` already pins `3.12` â€” keep it.
- Tighten `contracts/pyproject.toml`: `requires-python = ">=3.12,<3.13"` (was `>=3.12`) so the
  contracts env can never resolve onto 3.13+. The pin selects 3.12; the constraint forbids drift.
- `cd contracts && uv sync` must create `contracts/.venv` on **CPython 3.12** (uv auto-downloads it
  if absent â€” no system install needed) and generate `contracts/uv.lock`.

### 3. Route the build gate through the Makefile (stack-scoped, like the other three gates)
- Change `.ai-sdd/checks/changeset.build.check.yaml` command from `uv sync` to `make sync`.
- Add a `sync` target to the root `Makefile` that delegates to the stack:
  `sync:` â†’ `$(MAKE) -C contracts sync` (with commented stubs for operator/coprocessor/brain/pilot,
  same pattern as `test`/`validate`/`lint`).
- Add a `sync` target to `contracts/Makefile`: `sync:` â†’ `uv sync`.
- Net effect: "build" now means "sync **this stack's** env on **this stack's** Python," instead of a
  raw root `uv sync` that assumed a shared environment.

### 4. (Optional, document only) other stacks
- Do NOT scaffold operator/pilot/coprocessor here â€” they are out of this feature. But the root
  Makefile's commented stubs and this brief record the pattern they must follow: each gets its own
  `pyproject.toml` + `.python-version` (3.12 or 3.11 per the registry) + isolated `.venv`, and wires
  its own `sync`/`test`/`validate`/`lint` targets. No stack is ever a uv workspace member.

## Acceptance

1. Root `pyproject.toml` has NO `[tool.uv.workspace]` stanza (and is deleted entirely if it has no
   other purpose).
2. Root `uv.lock` is removed.
3. `contracts/pyproject.toml` has `requires-python = ">=3.12,<3.13"`.
4. `cd contracts && uv sync` exits 0 and `contracts/.venv` runs CPython **3.12.x**
   (`contracts/.venv/pyvenv.cfg` shows `version_info = 3.12.*`).
5. `contracts/uv.lock` exists.
6. `.ai-sdd/checks/changeset.build.check.yaml` command is `make sync`.
7. Root `make sync` exits 0 and delegates to `contracts`.
8. All four FIXED root gates still pass via the new routing: `make sync` (=build), `make test`,
   `make validate`, `make lint` â€” all exit 0.
9. `from contracts import ContractLine` still works after the standalone sync; the 15-test suite and
   the validate gate still pass (no regression from `pydantic-models` / `fixtures-and-gates`).

## Files in scope

## Notes / risks
- **Changing a global gate mid-run**: `changeset.build` is factory-wide. After this slice, every
  future slice's build gate routes through `make sync`. That is the intended end-state (stack-scoped
  builds) and is consistent with how `test`/`validate`/`lint` already route. The change is
  self-consistent within this slice: when this slice submits, the engine re-reads the check and runs
  `make sync`, which passes because the root + contracts Makefiles now define it.
- The interpreter swap from 3.14 â†’ 3.12 is the whole point; verify the contracts `.venv` actually
  reports 3.12 (don't just trust exit 0).
