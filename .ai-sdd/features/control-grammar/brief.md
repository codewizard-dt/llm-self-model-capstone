> **Scope-shaping precondition.** Four constraints from MASTER + the post-PR #18 catalog + the existing Brain sketch fix F19's shape before any decisions are open: (1) the command set is **fixed** ("a fixed command vocabulary" — MASTER Components), so the model is a closed `Literal` discriminator + pydantic tagged union, not an open string; (2) ADR-19 makes the online loop **bounded + interruptible**, so every motion command carries a `ttl_ms` and a `stop` command is always legal (already drafted in PROTOCOL.md); (3) the wire is **newline-delimited JSON on the existing USB user port** that already carries `ContractLine` records (ADR-10 + BRAIN_INTERFACE §3.3), so the command/ack envelope must use a `type` discriminator distinct from any value `ContractLine` could carry and ack lines must be self-identifying so the Pi can demux a single stream; (4) the parts-catalog design space is now **3 mutually-exclusive builds** — claw / scoop / flywheel (effector-encoded `MotorAllocation`) — so F19's *manipulator* verbs must cover all three concrete motor topologies: the `claw_grasper` build's `arm+claw` motors, the `scoop` build's `arm`-only motor (scoop is passive), and the `flywheel` build's `flywheel`-only motor (no arm in that build). One verb per addressable manipulator motor — anything less makes a Generator-revised build undrivable by F21 without a contract amendment.

---

## Goal

Freeze the **Control Grammar** as the runtime source of truth for the fixed, typed command/ack vocabulary the online control loop (`pilot` → `brain`, F21) uses to drive the robot in real time. Ship pydantic v2 models that validate (a) a closed **command envelope** carrying a discriminated union of motion/manipulator primitives, (b) the **ack envelope** the Brain returns, and (c) a controlled **fault** vocabulary; extend the contracts package's `schema` and `validate` entrypoints **additively** (no F1/F2/F3/F4 regression); and include fixtures for one cycle of each command kind + its expected ack. This is the artifact the `pilot` LLM emits, F20's PROS C++ bridge parses-and-clamps, F21 sequences against live telemetry + vision, and — per the ROS2 research recommendation — any future ROS bindings transport unchanged. **Closing this contract is what makes ADR-19 buildable**: F20 and F21 cannot start until F19 freezes the wire. F19 gates `m1`.

## In scope

- **Shared message envelope atom** — `contracts/src/contracts/control_command.py`:
  - `ControlEnvelope` (pydantic v2, `StrictModel` base, `extra="forbid"`) — the fields every Pi→Brain line carries: `v: Literal[1]` (protocol version, locked), `seq: int` (≥0; monotonic per-session, F19 only validates the type), `type: Literal["cmd", "heartbeat"]` (the **stream discriminator** — F19 freezes this so the Brain demux can route `cmd`/`heartbeat`/`ack` against the existing `ContractLine` stream on the same user port — see D2), `sent_ms: int` (Pi wall-clock at emit), `ttl_ms: int` (Field with bounds — see D5).
  - `ControlCommand` — the closed discriminated union of motion primitives, keyed on a `cmd: Literal[...]` discriminator on each member. **Frozen command vocabulary (D1, derived from the draft PROTOCOL.md, the Brain sketch's `MAX_*` constants, and the manipulator gap called out by BRAIN_INTERFACE §3.4 + F8's grab/pull/throw primitives):**
    - `StopCommand` — `cmd: Literal["stop"]`, `reason: Literal["operator_estop", "operator", "watchdog_self", "fault"]` (controlled vocabulary; F20 latches `operator_estop`).
    - **Drivetrain verbs (shared by all 3 builds — every config has the same 2-drive-motor `front_omni+rear_standard` topology per the post-PR #18 narrowing, where `wheel_config` is now a fixed kit reality rather than a vocabulary axis):**
    - `DriveCommand` — `cmd: Literal["drive"]`, `vx: float` (m/s, clamped `[-MAX_LINEAR, +MAX_LINEAR]`), `vy: float = 0.0` (locked to 0 because every buildable config uses the same `front_omni+rear_standard` drivetrain; field kept on the wire for forward-compat with a future holonomic chassis, but rejected if non-zero — D12), `omega: float` (rad/s, clamped `[-MAX_OMEGA, +MAX_OMEGA]`).
    - `TurnCommand` — `cmd: Literal["turn"]`, `omega: float` (rad/s, clamped). Kept distinct from `drive` (as in the draft PROTOCOL.md) so the `pilot` LLM can emit a pure-rotation intent the Brain can route through a different inner loop.
    - **Manipulator verbs (one per addressable manipulator motor across the 3 builds — see "Scope-shaping precondition" above and D1 below):**
    - `ArmCommand` — `cmd: Literal["arm"]`, `deg: float` (absolute target angle, clamped to a documented arm range), `vel_rpm: float | None = None` (optional speed cap; Brain falls back to its default if `None`). Drives the **arm-lift motor** present in the `2drive+1arm+1claw` (claw_grasper) build *and* the `2drive+1arm` (scoop) build — the scoop is passive, so its only manipulator motor *is* the arm-lift. Resolves PROTOCOL.md §5's open question and BRAIN_INTERFACE §3.1's "first real command will more likely be an arm action."
    - `ClawCommand` — `cmd: Literal["claw"]`, `state: Literal["open", "close"]`, `grip_force_N: float | None = None` (optional torque cap; reuses F1's `grip_force_N` unit so the Generator's `predicted.grip_force_N` and a commanded grip are in the same units, key-for-key with the F1 contract). Drives the claw motor in the `2drive+1arm+1claw` build only — maps to F8's `grab` primitive end-to-end with the `claw_grasper` end effector + `200rpm` cartridge (R4 `CLAW_CARTRIDGE` pins it).
    - `FlywheelCommand` — `cmd: Literal["flywheel"]`, `rpm: float` (target spin RPM, clamped `[0, MAX_FLYWHEEL_RPM]`; `rpm: 0.0` = coast/stop, distinct from a hard `stop` which also kills the drivetrain). Drives the flywheel motor in the `2drive+1flywheel` build only — that build has **no arm motor**, so `arm` is structurally inapplicable there (rejected by F20 at the Brain, per C5). Pinned to the `600rpm` cartridge by R3 `FLYWHEEL_CARTRIDGE`.
  - `HeartbeatLine` — `type: Literal["heartbeat"]` envelope with no inner body; Pi liveness ping when no command is pending.
  - `ControlLine` — the **outer wire model**: a discriminated union over `type` of `(envelope + ControlCommand)` (for `type == "cmd"`) and `HeartbeatLine` (for `type == "heartbeat"`). The single thing the Pi serializes and the Brain parses per line.
  - `AckLine` — `type: Literal["ack"]` (third, mutually-exclusive value of the shared `type` discriminator — see D2), `v: Literal[1]`, `ack: int` (the `seq` it acknowledges; required), `state: Literal["ok", "rejected", "stale"]` (controlled — see D6), `recv_ms: int` (Brain `pros::millis()` at receipt), `fault: FaultCode | None` (`None` iff `state == "ok"`), plus optional lightweight bridge-health fields proven by the guarded V5 Brain bridge (`battery_mv`, `battery_pct`, `watchdog_age_ms`, `estop`, `motion_enabled`, `drive_ports_ok`, `motor_ports`). Streaming motor samples stay on telemetry/ContractLine (D3).
  - `FaultCode` — `StrEnum` of `{"malformed_json", "unknown_command", "ttl_expired", "watchdog", "estop_latched", "out_of_range", "oversized_packet"}` (every code the existing Brain sketch's `ack(seq, "rejected", "<fault>")` paths already use, plus `out_of_range` for clamps that fail validation rather than silently saturate — D7).
  - **Clamp constants** — exported numeric constants in the same module: wire-envelope limits `MAX_LINEAR`, `MAX_OMEGA`, `MAX_FLYWHEEL_RPM`, arm range, and `TTL_MS_MAX`, plus guarded Brain physical clamps `BRAIN_MAX_LINEAR` and `BRAIN_TTL_MS_MAX` (see D5/D10).
- **`contracts/src/contracts/__init__.py` extended additively** — add `ControlLine`, `ControlCommand`, `ControlEnvelope`, `AckLine`, `HeartbeatLine`, `StopCommand`, `DriveCommand`, `TurnCommand`, `ArmCommand`, `ClawCommand`, `FlywheelCommand`, `FaultCode`, and the clamp constants to imports and `__all__`. All existing exports unchanged.
- **`schema` entrypoint, extended additively** — mirror F2's as-shipped `schema.py` registry (the `{filename: model}` direct-file-write pattern): add two entries, `control_command.json` (for `ControlLine`) and `ack_line.json` (for `AckLine`). The existing `contract_line.json`, `self_model.json`, `score_contract.json` outputs must be **byte-identical** (regression-gated by `make schema-check`).
- **`validate` entrypoint, extended additively** — third dispatch class. `validate.py` currently round-trips `fixtures/*.jsonl` → `ContractLine`, `fixtures/self_model_*.json` → `SelfModel`, `parts_catalog.json` → `PartsCatalog` (+ buildability of each F2 fixture's `config`). Add:
  - `fixtures/control_command_*.jsonl` → each line dispatches by `type`: `"cmd"`/`"heartbeat"` round-trips through `ControlLine`; `"ack"` round-trips through `AckLine`. Validates that fault codes are non-null iff `state == "rejected"`.
  Existing dispatches unchanged.
- **Fixtures** in `contracts/fixtures/`:
  - `control_command_grab_cycle.jsonl` — a representative grab-primitive cycle for the **`2drive+1arm+1claw` (claw_grasper) build**: `heartbeat` → `drive` (approach) → `arm` (lower) → `claw` (close) → `arm` (lift) → `stop`, each followed by its corresponding `ack` line on the next row. Anchors the MVP grab demo (Gen 0 → Gen 2). Annotated so the Generator (F8) and the `pilot` planner (F21) can replay-test their planning loop against a deterministic transcript.
  - `control_command_flywheel_cycle.jsonl` — a representative cycle for the **`2drive+1flywheel` build**: `heartbeat` → `drive` → `flywheel` (spin up to target RPM) → `flywheel` (rpm=0 coast) → `stop`. Exercises the `FlywheelCommand` path so the contract is proven across all three builds at m1, not just claw.
  - `control_command_rejections.jsonl` — one rejected ack per `FaultCode` value (drives `make validate`'s exhaustive fault-coverage assertion — see Acceptance §5).
- **Tests** in `contracts/tests/test_control_command.py` — see Acceptance §5 for the full list.

## Out of scope

- **F20 Brain-side parser** (the PROS C++ that reads a `ControlLine`, clamps it, drives motors, emits `AckLine`, runs the watchdog task). F19 freezes the wire; F20 satisfies it.
- **F21 online control harness** (the `pilot` Rx pipeline that builds states from F4's `TelemetrySource`/`VisionSource` and emits commands). F19 is one of its inputs.
- **The single-port multiplex *strategy*** (BRAIN_INTERFACE §3.3 — tagged single-stream vs RS-485 second channel). F19 **enables** the recommended "tagged single-stream" path by making `type` a closed `Literal["cmd", "heartbeat", "ack"]` discriminator and giving `AckLine` a top-level `type` field (so a reader thread can demux on `type` alone, without "next line = ack" assumptions). The reader-thread/seq-map implementation lives in F20 (Brain) and the Pi bridge (F20/F21).
- **Transport selection** — USB serial 115,200 (ADR-10) is the MVP transport. RS-485 Smart Port is a Stage-2 swap "same JSON, different baud" (MASTER → Integration Boundaries). F19 is transport-agnostic.
- **Binary protocol-v2** (CRC framing, message-type bytes) — the ROS2 research recommends keeping JSON v1 until first ROS proof is stable. Out of scope for F19; future contract.
- **Adapter protocols** for command flow (a hypothetical `ControlSink` to mirror F4's `TelemetrySource`/`VisionSource`) — F21 may want one, but F19 freezes only the data, not the abstraction over the sink.
- **Motor-level commands** (per-motor voltage/velocity, PID gain tuning). The grammar is **task-level**: drive/turn/arm/claw/stop. Per-motor primitives would bypass F20's clamps + watchdog and contradict ADR-19's "bounded + interruptible."
- **ROS message/action definitions** — F19 is the canonical schema; ROS bindings (if/when they land) are transport adapters per the ROS2 research recommendation.
- **Authentication / signing / authorization** on command lines — single-machine USB; no threat model justifies it for the MVP.
- **`Subject`s, `reactivex` operators, scheduler choice** for the online loop — F4's coordination items C2/C4 already pin those to F17/F21, not F19. F19 is pure pydantic.

## Acceptance

Each item is independently runnable from `contracts/` (or repo root, which delegates):

1. `make sync` exits 0.
2. `make lint` exits 0 (`ruff check` + `ruff format --check`).
3. `make schema` exits 0 and writes `contracts/schemas/control_command.json` + `contracts/schemas/ack_line.json` (committed, non-empty, the `type`/`cmd`/`state`/`fault` discriminators surface as JSON-Schema `enum` arrays). `make schema-check` exits 0; `contract_line.json`, `self_model.json`, `score_contract.json` are **byte-identical** to their pre-F19 versions (regression guard for F1–F3). *(Note: `self_model.json` was already re-emitted by PR #18; F19 must not perturb that post-narrowing baseline either.)*
4. `make validate` exits 0 — round-trips `control_command_grab_cycle.jsonl`, `control_command_flywheel_cycle.jsonl`, and `control_command_rejections.jsonl` through `ControlLine` / `AckLine` per the `type` dispatch, and still validates the F1 `*.jsonl`, F2 `self_model_*.json`, F3 `parts_catalog.json` (+ F2 fixture buildability) paths.
5. `make test` exits 0, covering:
   - round-trip parse of all three control fixtures;
   - the discriminated union: a `cmd` line missing `cmd` field → `ValidationError`; a `cmd` line with `cmd: "fly"` (out of vocabulary) → `ValidationError`; a `cmd: "drive"` body missing `vx`/`omega` → `ValidationError`;
   - per-command range validation: `drive` with `vx > MAX_LINEAR` → `ValidationError` (F19 rejects *out-of-range* values rather than silently clamping — clamping is F20's job and ADR-19's safety contract); `drive` with `vy != 0` → `ValidationError` (no holonomic chassis in any buildable config — D12); `turn` with `omega < -MAX_OMEGA` → `ValidationError`; `arm` with `deg` outside the documented arm range → `ValidationError`; `flywheel` with `rpm < 0` or `rpm > MAX_FLYWHEEL_RPM` → `ValidationError`; `ttl_ms` outside `[1, TTL_MS_MAX]` → `ValidationError`;
   - envelope shape: `v != 1` → `ValidationError`; `type: "telemetry"` (collision with `ContractLine`) → `ValidationError` (closed `Literal`); unknown extra field on `ControlEnvelope` or any command → `ValidationError` (`extra="forbid"`);
   - ack invariants: `state: "ok"` with non-null `fault` → `ValidationError`; `state: "rejected"` with `fault: null` → `ValidationError`; `state: "stale"` is accepted; every `FaultCode` enum value appears at least once in `control_command_rejections.jsonl` (exhaustive-coverage assertion);
   - clamp-constant identity: `MAX_LINEAR`, `MAX_OMEGA`, `MAX_FLYWHEEL_RPM`, `TTL_MS_MAX`, the arm range bounds are importable from `contracts` and match the documented values bit-for-bit (one source — F20 will import these instead of redeclaring; C1);
   - cross-contract demux: a `ContractLine` JSON line round-trips through `ContractLine` and **fails** to validate as `ControlLine` (the `type` discriminator is mutually exclusive — proves the single-stream demux strategy is wire-level safe);
   - F1, F2, F3, F4 test suites still pass (regression guard).
6. `from contracts import ControlLine, ControlCommand, AckLine, FaultCode, MAX_LINEAR, MAX_OMEGA, MAX_FLYWHEEL_RPM, TTL_MS_MAX` resolves cleanly in `uv run python` after `make sync`.
7. Root `make sync` / `make validate` / `make test` / `make lint` / `make schema` (delegating into `contracts/`) all exit 0; F1/F2/F3/F4 suites still pass.
8. Manual reviewer check (the qualitative gate for `m1`): a hand-authored `grab` cycle round-trips through `ControlLine` + `AckLine` exactly the way the existing `pros_bridge/src/main.cpp` would emit it today (existing `stop`/`drive` cases) plus the manipulator primitives (`arm`, `claw`, `flywheel`) the **3 post-PR #18 buildable configs** address — i.e. F19 frozen unblocks F20 to extend the Brain handler across the full parts-catalog design space without re-litigating the wire.

## Constraints

- Python 3.12 (`requires-python = ">=3.12,<3.13"`) · uv · ruff · pydantic v2 — no pip/poetry/black/flake8 (ADR-05, ADR-15, ADR-16). Build entry is **`make sync`** (F1 D7).
- `root: contracts/` · `ignore_folders: .venv, __pycache__, dist, .pytest_cache, captures`.
- pydantic v2 only: `ConfigDict`, `Field`, discriminated unions via `Field(discriminator=...)`; JSON Schema via `model_json_schema()` (ADR-06). Reuse F1's `StrictModel` base for all strict sub-models.
- `v: Literal[1]` is locked; `schema_version` is not used on the wire (the protocol version `v` plays that role — D8); bumping `v` requires a new frozen-contract approval (parallels F1's `schema_version`).
- **Closed vocabulary**: every discriminator is a `Literal` / `StrEnum` — `type ∈ {cmd, heartbeat, ack}`, `cmd ∈ {stop, drive, turn, arm, claw, flywheel}`, `state ∈ {ok, rejected, stale}`, `reason ∈ {operator_estop, operator, watchdog_self, fault}`, `FaultCode ∈ {malformed_json, unknown_command, ttl_expired, watchdog, estop_latched, out_of_range, oversized_packet, not_assembled}`. **No** free-string `cmd` and **no** free-string `fault` — that is the whole point of "fixed command vocabulary" (MASTER) and ADR-19 ("fixed control-grammar commands"). The verb set spans the **3 post-PR #18 buildable motor topologies** (one verb per addressable manipulator motor) — adding a primitive is a frozen-contract change, the same way F1's typed-task expansion was.
- **Strict envelopes**: every command body + envelope + ack uses `extra="forbid"` so a malformed `pilot` emission fails fast at F19, not at the Brain.
- **No telemetry leakage on the ack** (D3): the ack carries `state` + `fault` only. Telemetry shape lives in F1.
- **Wire discriminator non-collision**: the `type` field's `Literal` values must not overlap with anything `ContractLine` puts on the same stream — this is enforced by the cross-contract demux test (Acceptance §5).
- **Additive only**: every existing model, schema file, fixture, and Makefile target is untouched. `schema.py`'s registry gains two entries; `validate.py` gains one dispatch.
- No hardware, no network, no schema defined outside `contracts`, no PROS C++ — F20/F21 own those.

## Decisions

| # | Decision | Resolution | Status |
|---|---|---|---|
| D1 | Command vocabulary (frozen at m1) | **`stop`, `drive`, `turn`, `arm`, `claw`, `flywheel`** — chosen to span the **post-PR #18 narrowed parts-catalog design space (3 buildable motor topologies)**: every config has a 2-motor `front_omni+rear_standard` drivetrain (→ `drive`/`turn`/`stop`); the `2drive+1arm+1claw` claw build adds `arm`+`claw`; the `2drive+1arm` scoop build adds `arm` only (scoop is passive — no second manipulator motor); the `2drive+1flywheel` build adds `flywheel` only (no arm motor in that build). One verb per addressable manipulator motor. The earlier scope-and-defer plan ("ship claw only, add scoop/flywheel when F8's primitives expand") was rejected after PR #18 because **the Generator can already revise across all 3 builds today** — deferring would leave 2/3 of the buildable design space un-drivable by F21 until an immediate contract amendment. Anything *more* (cartridge-aware speed presets, scoop-specific verbs that don't exist mechanically, holonomic `vy`) is still deferred; expanding the closed `Literal` remains a new frozen-contract change. | `closed` |
| D2 | Stream discriminator | **Closed `type: Literal["cmd", "heartbeat", "ack"]`** on every wire line, with `type` as the **single demultiplexer** for the Pi reader thread (BRAIN_INTERFACE §3.3 Option 1 — "tagged multiplexing recommended"). The cross-contract demux test (Acceptance §5) proves `ContractLine`'s payload cannot collide with any value in this set. | `closed` |
| D3 | Ack field set | **Receipt plus lightweight bridge health.** The required receipt remains `state` + `fault` + `ack` + `recv_ms`; optional `battery_mv`, `battery_pct`, `watchdog_age_ms`, `estop`, `motion_enabled`, `drive_ports_ok`, and `motor_ports` match the live guarded Brain ack. Streaming motor samples remain telemetry/ContractLine. | `closed` |
| D4 | Discriminator placement | **Two-level**: outer `type` (cmd/heartbeat/ack) at the envelope, inner `cmd` (stop/drive/turn/arm/claw) inside `cmd`-typed lines. pydantic v2 discriminated unions handle both. Lets a future protocol-v2 binary framing replace the outer envelope without touching the command bodies. | `closed` |
| D5 | TTL bounds | **`ttl_ms: int` with `Field(ge=1, le=TTL_MS_MAX=5000)`** — matches the ROS bridge envelope. The guarded Brain narrows motion TTL with `BRAIN_TTL_MS_MAX=500`, so ADR-19's bounded property is enforced both wire-side and at the hardware boundary. | `closed` |
| D6 | Ack `state` values | **`ok` / `rejected` / `stale`.** `stale` is added for the case where the Brain receives a command **after** its TTL has expired (a real failure mode under serial backpressure) — distinct from `rejected` (Brain refuses to act) and gives F20 a clean signal to log without re-stopping. | `closed` |
| D7 | Fault codes — closed enum or free string | **Closed `StrEnum`.** Free-string faults would force the Pi to do string parsing to route; a closed set lets F20's `ack()` builder and the Pi's fault handler both pattern-match exhaustively. The eight codes (`malformed_json`, `unknown_command`, `ttl_expired`, `watchdog`, `estop_latched`, `out_of_range`, `oversized_packet`, `not_assembled`) cover every reject path in the existing Brain sketch, the validation-failure case F19 introduces, plus `not_assembled` — the new code F20 emits when a verb is wire-legal but mechanically inapplicable to the *currently assembled* build (e.g. `flywheel` to a `2drive+1arm+1claw` robot, or `claw` to a `2drive+1arm` scoop robot). This **decouples wire-legality from runtime-applicability** (C5): F19 freezes the full cross-build vocabulary so the Generator/`pilot` can plan against any of the 3 builds; F20 enforces what the *physical* robot in front of it can actually do. | `closed` |
| D8 | `schema_version` vs `v` | **Use `v: Literal[1]`** on the wire (matches the draft PROTOCOL.md and the existing Brain handler). No `schema_version` field on `ControlLine`/`AckLine` — `v` is the version. F1's `schema_version` lives on `ContractLine` and stays there; the two contracts version independently (mirrors F2/F3 D11). | `closed` |
| D9 | Module + schema file names | **`contracts/src/contracts/control_command.py`** (mirrors `contract_line.py`/`self_model.py`/`parts_catalog.py` naming); schemas emit to `contracts/schemas/control_command.json` (the command/heartbeat union) and `contracts/schemas/ack_line.json` (the ack). Two files because they validate different lines on the same stream — a single combined schema would force every consumer to encode the demux. | `closed` |
| D10 | Clamp constants — where do they live | **Exported from `contracts.control_command`**: wire-envelope constants (`MAX_LINEAR`, `MAX_OMEGA`, the arm range, `TTL_MS_MAX`) plus guarded Brain clamps (`BRAIN_MAX_LINEAR`, `BRAIN_TTL_MS_MAX`). F19 enforces wire bounds and exposes the narrower hardware clamps for F20. | `closed` |
| D11 | Out-of-range = reject or clamp at F19? | **Reject (`ValidationError`).** F19 is the *contract* — a value outside the documented range is a `pilot` bug, not a wire feature. F20 still clamps at the Brain (its safety contract: never trust the Pi), but the Pi-side parser should never silently saturate a malformed emission. | `closed` |
| D12 | `DriveCommand.vy` — drop, keep at 0, or accept | **Keep on the wire, locked to 0** (`Field(ge=0, le=0)` — a non-zero `vy` is a `ValidationError`). The post-PR #18 catalog made `wheel_config` a fixed kit reality (every buildable config uses `front_omni+rear_standard`), so a holonomic `vy` is structurally inapplicable. Keeping the field present means a future kit upgrade to a holonomic drivetrain is a wire-relaxation (drop the `le=0` bound) rather than a wire addition; dropping the field would make that a breaking change. Aligns with the published PROTOCOL.md draft which already lists `vy`. | `closed` |
| D13 | Cross-build verb applicability — where enforced | **F20, not F19.** F19 freezes the verb set across all 3 buildable motor topologies (any verb is wire-legal). The currently-assembled build is a *runtime* fact (which motors are plugged into which ports) that F19 cannot see from a static JSON document — enforcing it at F19 would require either embedding `motor_allocation` in every command (wire bloat + redundant — the robot already knows what it is) or a stateful contract (anti-pattern). F20 rejects mechanically-inapplicable verbs with `FaultCode.not_assembled` (D7). | `closed` |

## Open questions

- **O1** *(scope-bounding — proposes a closure)* The **arm angle range** for `ArmCommand.deg`. Today's `v5-test/src/main.cpp` writes a 300° arm-raise as a free numeric; there is no documented mechanical range for the Clawbot arm in this repo. The arm motor is shared between the `2drive+1arm+1claw` (claw_grasper) and `2drive+1arm` (scoop) builds, which may have different effective ranges due to the scoop's mass/geometry. Proposed default: `Field(ge=0, le=360)` with the **operational** sub-range tightened in F20 per build once the team measures the assembled hard stops. F19 ships with the conservative `[0, 360]` bound; F20 narrows per build. (Could also be deferred entirely to F20 — needs an owner call.)
- **O1b** *(new — opened by PR #18)* The **`MAX_FLYWHEEL_RPM` value**. The catalog pins flywheel to the `600rpm` cartridge (R3), so the *cartridge-rated* spin ceiling is ~600 RPM at the output shaft, but the flywheel itself may spin faster than that depending on the launch mechanism's gearing — and the existing repo has no flywheel hardware measurement yet. Proposed default: `MAX_FLYWHEEL_RPM = 3600` (the V5 11W Smart Motor's free speed at the 600rpm cartridge accounts for plenty of gear-up; better to be permissive at the wire and let F20's clamp + the motor's hardware PID enforce reality). Confirm with the flywheel-build owner.
- **O2** Whether F19 should ship a tiny generated **C++ header** for F20 to import the clamp constants (C1's "pragmatic path"). Lower-risk alternative: F20 reads `contracts/control_command.py` constants once at codegen-time and inlines them, with a `make check` diff guard. Decision deferred to the F20 owner.
- **O3** Whether to add a **`reason: str`** free-text field to `StopCommand` alongside the controlled `reason` enum (the draft PROTOCOL.md has free-text `reason: "operator"`). The current decision (D7) is *closed enum, no free text* — re-open only if F21 needs to log human-readable intent that can't be expressed by the four enum values.
- **O4** Whether `seq` should be `Field(ge=0)` or carry a wraparound invariant (`uint32`). Today's Brain sketch uses `std::atoi` and emits acks with whatever `seq` arrived. Proposed: `Field(ge=0, lt=2**32)` to match a future binary framing; deferred until F20 owner weighs in.
- **O5** *(MASTER reconciliation)* MASTER's ADR-19 notes "on-device online inference likely needs an API key + network (contradicting 'no keys'); the runtime + secret posture for `pilot` is an open decision." That decision affects F21, **not** F19 — the wire is unchanged whether `pilot` runs on Claude Code or an API. Flagged for transparency; F19 is decoupled and can freeze independently.
