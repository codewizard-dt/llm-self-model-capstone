Grace · premise

Real body, readable self-model.

Grace makes the premise concrete for the room: Vexy is not an abstract AI demo, it is a real VEX V5 robot whose physical body, sensors, claw, yellow ball task, and language self-model make the capstone technically real.

  1. Open with the simplest truth: a robot that can be wrong can learn.
  2. Name the premise exactly: Self-Modeling Machines, in Language.
  3. Explain that Vexy is the real robot body: VEX V5 Brain, Smart Ports, Smart Motors, metal rails, wheels, red claw, and battery-constrained hardware.
  4. Add the perception stack: Raspberry Pi, Camera Module 3, AprilTag localization, and the yellow ball mission sample.
  5. Define self-modeling in plain English: Vexy needs an internal account of what its own body is and what it can do.
  6. Explain the language twist: the model should be readable enough for humans to challenge, not only an opaque tensor.
  7. Connect the run: Vexy predicts in language, acts around the yellow ball and tag, then lets reality prove it right or wrong.
  8. Name Loop 1 calibration and Loop 2 future morphology redesign.
  9. Make the primitive arc memorable: grab, pull, throw.
  10. Handoff to David: once the audience understands the body, David explains what learning means as a loop.

Simple note card

Self-Modeling Machines, in Language means Vexy can compare its imagined body to what reality measures.
  • Vexy is a physical VEX V5 robot with a Raspberry Pi vision stack.
  • The yellow ball and AprilTag make the proof run concrete.
  • The goal is Sense, Act, Measure, Improve.

Closing line: Once the body is real, David can explain how a run becomes a learning loop.

David Taylor · learning loop

Learning is a loop, not a claim.

David Taylor explains the research leap: the small loop is generate a plan, execute it, analyze telemetry, and start over; the larger Generational Loop asks how an LLM could eventually redesign its own body from real-world evidence.

  1. Open by separating truth from hype: the small learning loop is not "LLM says robot improved."
  2. Name the sequence: generate a plan, execute it, analyze telemetry, and start over.
  3. Explain the Generational Loop: an LLM could eventually redesign its own body.
  4. Ground the idea in numerical self-modeling and the sub-symbolic tensor limitation.
  5. Ground it in LLM morphology generation and the simulation limitation.
  6. Ground it in LLM self-improvement and the missing self-model component.
  7. Name the novelty: readable language self-modeling plus telemetry plus future morphology.
  8. Name evidence sources: motor samples, vision state, operator results, and run outcome.
  9. Stay honest about simple tasks and ball-in-bin progress.
  10. Handoff to Erick: safe motion has to come before trusted learning.

Erick · trusted motion

The hard part is trusted motion.

Erick explains the hard engineering problem: moving a robot is one layer, but trusted motion requires bounded commands, strict contracts, ack/fault states, watchdogs, and evidence from the run.

  1. Open with: making the robot move is one problem; making it move safely, measure itself, and explain what changed is harder.
  2. Name wheel slip, motor drift, camera errors, object ambiguity, claw alignment, AprilTag angle, and a moving ball.
  3. Say the problem is not motion or action; the problem is trusted motion.
  4. Explain why the LLM cannot directly freestyle against motors.
  5. Explain bounded vocabulary and safe command packets.
  6. Name contract states: accepted, rejected, stale, completed, or faulted.
  7. Explain watchdogs and limits as bounded-run guardrails.
  8. Connect contracts to learning: no evidence trail means no safe learning.
  9. Close with motion that can be trusted, measured, and explained.
  10. Handoff to Jake: bounded motion becomes evidence.

Simple note card

  • Ack/fault states and evidence make motion measurable.

Jake · evidence packet

The gap is the learning signal.

Jake explains the measurement layer: each run needs prediction, observation, telemetry, vision state, gap, and update so the system can learn from evidence instead of vague narration.

  1. Open with measurement: this is where the project becomes more than a robotics demo.
  2. Define the evidence packet: predicted, observed, telemetry, vision, gap, update.
  3. Use the yellow ball: if the claw is off, the miss becomes evidence.
  4. Explain AprilTags and vision as sensing before action.
  5. Explain predicted versus observed in plain English.
  6. Name the gap as the learning signal.
  7. Structure keeps the LLM from inventing explanations.
  8. Connect to the critic gate before trusting the model update.
  9. Close: Vexy treats every run as evidence.
  10. Final handoff: body, learning loop, trusted motion, and evidence become the mission.

Simple note card

A run matters only if it leaves behind evidence the model can inspect.
  • Prediction and observation have to be compared.
  • Telemetry and vision state keep the story grounded.
  • The gap is the learning signal.

Closing line: Together, the team showed how Vexy can move from body to model to proof.
